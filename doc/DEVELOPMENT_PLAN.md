---

## 一、项目总览

| 维度 | 内容 |
|------|------|
| **产品定位** | 本地优先的 AI 原生数字资产管理平台（DAM） |
| **技术栈** | Electron 28+ / React 18 + TS / SQLite(FTS5) / Drizzle ORM / TailwindCSS + Arco Design |
| **目标平台** | Windows 10+ → macOS 12+ → Linux |
| **核心差异化** | AI 自动标签 / 以图搜图 / 浏览器一键采集 / 设计软件原生插件 |

| 包管理 | pnpm + workspace | 快速 + 严格依赖 |

**技术要点**:

```typescript
// 导入流程设计
文件选择 → [验证] → [元数据提取] → [缩略图生成] → [DB写入] → [UI刷新]
              ↓          ↓               ↓             ↓
          格式黑名单  exifr库         Sharp流水线    批量INSERT
                      ffmpeg(视频)    视频抽帧       事务保证
```

**性能基准达成目标**:
- 导入速度 > 500 文件/秒
- 缩略图生成 < 100ms/张（已缓存时）
- 内存占用稳定，无泄漏

**搜索架构实现**:

```
用户输入 → 防抖(300ms)
         → 分词处理
         → ┌─ FTS5全文匹配 (file_name/description/source_url)
         ├─ 元数据过滤 (type/folder/color/size/date)
         └─ 标签联合查询 (asset_tags JOIN)
         → 结果融合排序 → 分页(每页60条) → 虚拟滚动渲染
```

## 七、关键技术风险与应对

| 风险 | 概率 | 影响 | 应对策略 |
|------|------|------|----------|
| Electron 安装包过大 (>150MB) | 高 | 高 | 按需加载 Native 模块；Tauri 作为备选方案 |
| AI 本地推理效果不佳 | 中 | 中 | 预留 OpenAI API fallback 通道 |
| SQLite 大数据量下性能衰减 | 低 | 高 | WAL 模式 + 分库分表策略 + 定期 VACUUM |
| 虚拟滚动在复杂布局下闪烁 | 中 | 中 | 采用 key 复用策略 + 占位高度动态校正 |
| 跨平台文件路径兼容问题 | 高 | 中 | 统一抽象层 `PathService`，单元测试覆盖 |
| FTS5 中文分词效果差 | 中 | 高 | 引入 jieba 分词或使用 external content FTS5 方案 |
| 浏览器扩展 NativeMessaging 权限限制 | 中 | 中 | 提前研究各浏览器安全策略，准备降级方案（HTTP Bridge） |
